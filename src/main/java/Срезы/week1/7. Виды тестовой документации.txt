Виды тестовой документации

1. тест-план
    Тест план (Test Plan) - это документ, описывающий весь объем работ по тестированию, начиная с описания
    объекта, стратегии, расписания, критериев начала и окончания тестирования, до необходимого в процессе
    работы оборудования, специальных знаний, а также оценки рисков с вариантами их разрешения.
    тест план включает в себя разделы:
        • цель
        • Объекты тестирования, которые будут/не будут тестироваться
        • тестовая стратегия
        • Критерии (начала, приостановки, возобновления, завершения) тестирования
        • Ресурсы
        • расписание
        • роли и ответственность
        • оценка рисков
        • Документация
        • Метрики

2. тестовая стратегия
    Тестовая стратегия определяет то, как мы тестируем продукт. Это набор мыслей и идей, которые направляют
    процесс тестирования.
    В стратегии тестирования описывают:
        • Тестовую среду.
        • Анализ рисков проекта.
        • Инструменты, которые будут использовать, чтобы провести автоматизированное тестирование и для других
        целей.
        • План действий при непредвиденных обстоятельствах.

Различие плана и стратегии
    Тест-план:
        • устанавливает цели процесса тестирования;
        • определяет, что будет проверяться.
    Стратегия тестирования:
        • описывает, как достичь целей, поставленных в плане тестирования.

3. чек-лист
    набор идей по тестированию, разработке, планированию и управлению.

    Чек лист обязательно должен содержать в себе следующую информацию:
        • идея проверок;
        • набор входных данных;
        • ожидаемые результаты;

    Чек-лист должен обладать рядом важных свойств:
        • Логичность.
        • Последовательность и структурированность.
        • Полнота и неизбыточность.

4. Test Case (тестовый сценарий)
    набор входных данных, условий выполнения и ожидаемых результатов для проверки.

    Уровни тест-кейсов:
        - высокоуровневый: не содержит конкретных входных данных и ожидаемых результатов
        - низкоуровневый: содержит конкретные входные данные и ожидаемые результаты

        содержание тест-кейса:
            - номер
            - заголовок
            - предусловия
            - шаги проверки
            - ожидаемый результат
            опц
            - статус
            - модуль
            - приоритет

        Статусы тест-кейса:
            - passed
            - failed
            - skipped
            - no run
            - blocked

        Правила составления:
            - независимость
            - однозначность
            - полнота
            - 1 кейс = 1 цель
            - обезличенность

5. тест-сьют
    совокупность тест-кейсов, выбранных с некоторой общей целью или по некоторому общему признаку.

6. пользовательская история (User Story)
    способ описания требований к разрабатываемой системе, сформулированных как одно или более предложений на
    повседневном или деловом языке пользователя. Пользовательские истории используются гибкими методологиями
    разработки программного обеспечения для спецификации требований.
    Иными словами, User Story — это короткая формулировка намерения, описывающая что-то, что система должна
    делать для пользователя. Пользовательская история фиксирует короткую формулировку функции на карточке или,
    возможно, с помощью онлайн-средства.

7. отчет о дефекте (Bug report)
    документ описывающий ситуацию, которая привела к обнаружению дефекта

    Дефект - несоответствие фактического результата ожидаемому.

    Элементы Bug report:
    - идентификатор
    - краткое описание
    - подробное описание
    - шаги воспроизведения
    - фактический результат
    - ожидаемый результат
    - приложения
    - Критичность (severity - важность для проекта): blocker, critical, major, minor, trivial
    - приоритет (priority очередность устранения): high(блокирует), medium(критические), low(низкие)
    - окружение
    - статус
    опц
    - возможность обхода
    - воспроизводимость
    - комментарии

8. Жизненный цикл Bug report:
    - обнаружен -> назначен -> исправлен -> проверен
    - может иметь переходные состояния: отклонен, открыт заново

9. Отчет о тестировании, содержание:
    - команда тестировщиков
    - описание процесса тестирования
    - краткое описание
    - расписание
    - рекомендации
    - статистика по ошибкам

Комментарии по итогу фин среза:
    Что проще и дешевле написать - чек лист
    Что сложнее поддерживать - тест-кейсы (содержат много детальной информации)
    Что новичку проще принять/работать - теcт-кейсы (потому что прописаны шаги)
